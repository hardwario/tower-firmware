## Findings
- **FINDING-1 (test bug, edge_frame_limits):** asserts frame type 7 = "unknown" → BadType, but `FrameType::Beacon = 7` was added; `from_u8(7)=Some(Beacon)`, so a tampered-to-7 header is valid-type but fails CCM → AuthFail. Test should use type ≥8 (e.g. `|0x08`). Frame codec is correct.
- **FINDING-2 (test bug, radio_regdump):** CHFLT (0x1D) expected 0x23, but config.rs:275 intentionally writes 0x02 (~216 kHz RX BW, per docs/radio.md). Radio is correct; regdump's expected value is stale → should be 0x02.
- **FINDING-3 (observability, console):** multi-line one-shot KATs (>~6 lines) emit `info!` synchronously without yielding; the depth-8 drop-newest TX queue (src/console.rs TX_DEPTH) drops later lines INCLUDING the verdict, before the writer task runs. This masked FINDING-1. Affects edge_frame_limits (13 lines) most. Fix options: yield between lines in such examples, or raise TX_DEPTH (RAM-bound: 64 overflows fhss_compliance RAM).

## RESOLUTION (follow-up)
- FINDING-1 FIXED: examples/edge_frame_limits.rs — type 7→8 + quiet-on-success. HW: `ALL PASS ***` at TX_DEPTH=8.
- FINDING-2 FIXED: examples/radio_regdump.rs — **4** stale expectations corrected (CHFLT 0x23→0x02, PCKTCTRL2 0x87→0x1F, PCKTCTRL1 0x60→0x70, PROTOCOL2 0x00→0x02; last 3 were hidden by the queue until #3 fixed) + per-line pacing. HW: 23 regs, 0 MISMATCH at depth 8. (config.rs values confirmed; radio link works.)
- FINDING-3 FIXED at example level (no RAM cost), TX_DEPTH kept 8: edge_frame_limits logs only failures+verdict; radio_regdump paces ~8ms/line so the writer drains. Both fully observable at depth 8.
- FINDING-4 RE-DIAGNOSED — NOT tower-cli: `tower monitor` shows the device emits Dropped payload `e8ffffff0f` = 4294967272 (= u32 −24) verbatim; tower-cli displays it faithfully. Firmware DROPPED is saturating_add(1) (can't reach −24) and 4B drops between adjacent frames is impossible → memory corruption. Reproduced ONLY on net_bulk_stress (monolithic 4 KB stack buffer, documented stack-tight); net_bulk_stream (streaming) is clean. Root cause = net_bulk_stress stack pressure corrupting the .bss DROPPED counter; transfer CRC still passes. tower-cli left untouched.

## FINDING-4 RESOLVED (firmware, net_bulk_stress)
Stack-painting on HW: peak stack ≈ 9.0 KB (BLOB_LEN-independent — the `out` buffer is in the
task future/.bss, not the stack). free-stack-at-peak = 4044 − BLOB_LEN B. So BLOB_LEN=4096 →
free_at_peak = −52 B: the stack overruns the top of .bss by ~50 B and clobbers a static (the
console DROPPED counter → garbage `4294967272` drop count); the transfer's CRC still passes.
Fixed: BLOB_LEN 4096→2048 (free_at_peak ≈ 1996 B, ~2 KB margin) + corrected the example's
limit comment with the measured model. HW-verified: rounds 6→16 OK, 0 drop-count glitches over
65 s. tower-cli untouched (it faithfully displayed the device's count).
